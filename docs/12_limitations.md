# 12. Ограничения и риски

> Не самокритика ради самокритики - честная карта слабых мест, чтобы на защите
> быстро и спокойно ответить «что у вас еще не идеально».

---

## 12.1 No-answer queries пока не идеальны

Самое заметное слабое место - вопросы, на которые в чате на самом деле нет ответа.

### Почему сложно

- Retrieval почти всегда находит «что-то похожее» - space не пустой.
- Без отдельной abstention-логики система склонна вернуть хоть какие-то `message_ids`.
- Текущий harness (`eval/metrics.py`) на пустом `relevant` возвращает `0`,
  поэтому даже «правильное молчание» не награждается.

### Что дальше

- Threshold на score top-кандидата → `[]` при низкой уверенности.
- Стратегия «лучше пусто, чем уверенно ошибиться».
- Отдельный negative-набор вместо текущего eval.

---

## 12.2 Парафразы, опечатки, морфология - всё ещё хрупко

Dense + enriched question покрывают большую часть, но остаются слепые пятна.

### Особенно рискованно

- Очень разговорный вопрос, а в чате - техническая формулировка.
- Почти-совпадающие технические токены (`SDK 3.2` vs `sdk3.2` vs `sdk-3.2.0`).
- Склонения (`митапа / митапе / митапом`).
- Опечатки и разные варианты имени (`macbook` / `MacBook` / `мак`).

### Что дальше

- Морфологическая нормализация query tokens (pymorphy2 или аналог).
- Словарь алиасов для имён, документов, продуктов.
- Query expansion через синонимы (осторожно - легко уронить precision).

---

## 12.3 Multi-message ответы требуют аккуратной сборки

Если правильный ответ занимает несколько сообщений, задача превращается в
«найти + упорядочить», а это сложнее single-hit retrieval.

### Где заметно

- Длинные технические треды с уточнениями и follow-up'ами.
- Анонсы из нескольких сообщений.
- Запросы «весь тред», «все сообщения по теме».

### Что дальше

- Thread-aware scoring: буст за покрытие того же `thread_sn`.
- Cluster-level coverage: если нашли 3 из 5 сообщений треда, добавить остальные.
- Отдельные тесты именно на multi-message ranking.

---

## 12.4 Metadata (`thread_sn`, date) не всегда надёжны

В теории chat metadata сильно помогает. На практике:

- `thread_sn` может быть пустой (плоские чаты).
- Date metadata не всегда выразительна (месяцы сжаты в «недавно»).
- По `participants` тоже не всегда безопасно фильтровать жёстко - можно потерять релевантное.

Поэтому мы сознательно делаем ставку на soft signals (boost `+0.06`),
а не на hard filters.

Плюс: меньше риск потерять recall.
Минус: precision не идёт выше определённого потолка - часть FP мы не отсекаем.

---

## 12.5 Зависимость от внешнего dense API и reranker

Не баг, а операционный риск.

### Что может пойти не так

- `429 Too Many Requests` под нагрузкой.
- Сетевые задержки / timeout.
- Временная недоступность upstream.
- Нестабильное время ответа rerank'а.

### Что уже закрыто

- In-memory cache (dense + rerank).
- `httpx.AsyncClient` с retry.
- Fallback на sparse-only retrieval при dense-фейле.
- Fallback на retrieval order при rerank-фейле.
- Возврат `200 status=degraded`, а не `500`.

Проверено `scripts/chaostest.py` - все сценарии PASS.

### Что дальше

- Warm-up при старте (pre-flight embedding самых частых запросов).
- Offline latency profile (p50/p95 каждого upstream'а отдельно).
- «Защитный» режим демо - sparse-only + отключённый rerank.

---

## 12.6 Локальный eval ≠ hidden test

Это важно проговорить на защите честно.

### Что локальный eval даёт

- Быстрый цикл проверки гипотез.
- Разбор конкретных промахов (`--verbose --stages`).
- Chaos/load test против реальных контейнеров.

### Чего локальный eval не даёт

- Гарантии, что hidden-тест устроен так же.
- Покрытие редких типов вопросов.
- Representative сэмпл no-answer queries.

### Риски hidden-теста

- Другой тип чатов (не наш локальный анонимизированный чат).
- Другой баланс технических / нетехнических вопросов.
- Больше negative-кейсов (пустые `message_ids`).
- Другие формулировки enriched question (может не быть `hyde`/`variants`).
- Иная доля вопросов, где критичны даты, авторы, документы.

---

## 12.7 Логика всё ещё эвристическая

Нормально для хакатона, но назвать честно.

Система работает благодаря:
- chunking;
- enriched question целиком;
- hybrid retrieval;
- нескольким слоям ранжирования.

Но часть логики - ручные коэффициенты:
- `RERANK_ALPHA = 0.2`;
- phrase boost `0.04…0.07`;
- `context_penalty = 0.08`;
- cap'ы (`min(..., 0.24)`);
- лимиты prefetch / rerank.

### Что дальше

- Систематический sweep по группам параметров (не только chunker).
- Ablation-таблица: выключать по одному компоненту и мерить delta.
- Возможно, lightweight learning-to-rank слой вместо ручных boost'ов.

---

## 12.8 Главный следующий шаг

Если выбирать одно-два реально сильных улучшения (а не «десять красивых идей»):

1. Abstention logic для no-answer queries.
2. Multi-message ranking для длинных тредов.
3. Systematic ablation вместо точечных sweep'ов.

---

## 12.9 Честная формулировка для защиты

> Архитектура уже рабочая. Рост мы видим не в радикальной
> перестройке, а в двух точках: no-answer queries и более тонкий ranking
> длинных многосообщных ответов. Остальное - тонкий тюнинг коэффициентов,
> который без ablation-таблицы будет плясками с бубном.

---

## 12.10 Примеры — где именно ломается система

Для каждого ограничения — конкретный запрос, который его демонстрирует.

### 12.10.1 No-answer query (§12.1)

```json
{"question": {"text": "кто написал драйвер для квантового компьютера в чате?"}}
```

В локальном анонимизированном чате такого обсуждения не было.

Что делает система сейчас:

```json
{
  "message_ids": [
    "4555555555555555555",
    "4555555555555555556",
    ...
  ],
  "status": "ok"
}
```

Retrieval всё равно вернул ближайшие по dense-similarity сообщения про «драйвер»
или «компьютер». Metric по такому вопросу с пустым ground truth = 0.

Что нужно: threshold на top-score — если максимальный `blended_score < 0.4`,
возвращать `[]`. Но нужен отдельный negative-набор для калибровки.

### 12.10.2 Парафраз + морфология (§12.2)

```json
{"question": {"text": "на чём сломался десктопный клиент у Бобрика?"}}
```

Правильный ответ — тред про `SIGABRT на MacBook Air M1` от пользователя «Бобр».

Где ломается:

- «десктопный» → dense модель понимает, sparse — нет.
- «клиент» → не совпадает ни с `desktop`, ни с `runtime`, ни с `SIGABRT`.
- «Бобрика» → склонение, не матчится с `Бобр` в `participants`.

Что спасает (частично): HyDE-pass. Если в `question.hyde[]` есть
`"десктопный клиент упал с ошибкой"`, dense-ветка вытянет правильный тред. Но если
enriched-question бедный — промах.

### 12.10.3 Multi-message ответ (§12.3)

```json
{"question": {"text": "расскажи весь тред про релиз SDK 3.4?"}}
```

Правильный ответ — 8 сообщений из треда `thread_sn=t17`.

Что происходит:

- Retrieval вытягивает 3-4 chunk'а с этим тредом.
- Внутри каждого chunk'а `score_message_block` оценивает message-level score.
- Из 8 сообщений топ-блоки — `msg_12, msg_14, msg_15` (с самыми плотными hit'ами).
- `msg_13, msg_17` (follow-up'ы без явных ключевых слов) получают низкий block_score
  и могут не попасть в top-50.

Delta на метрике: Recall@50 по таким вопросам обычно 0.62-0.75, тогда как
по «single-hit» вопросам 0.95+.

### 12.10.4 Metadata не помогает (§12.4)

```json
{"question": {
  "text": "что обсуждали 12 апреля?",
  "date_range": {"from": "2024-04-12", "to": "2024-04-13"}
}}
```

Что ожидали: hard filter по датам → только сообщения 12 апреля.

Что делаем сейчас: soft boost `+0.06` для chunk'ов, чей `[start, end]`
пересекает range. Но вопрос «что обсуждали» широкий — retrieval возвращает
много «похожих по смыслу» chunk'ов вообще без пересечения по датам, и они
вытесняют нужные.

Почему не делаем hard filter: если `date_range` приходит неточный
(`from: 2024-04-10, to: 2024-04-15`, а ответ — 12 апреля, попадает), всё ок.
Но если enriched-question пропустил день / сдвинул на неделю — hard filter
режет правильный ответ. Soft boost «прощает» ошибку, hard — нет.

### 12.10.5 Upstream dependency (§12.5)

Ситуация: reranker API отвечает `429 Too Many Requests` — хакатонный limiter
сработал при параллельных evaluation-запросах.

Что делает система:

```json
{
  "message_ids": [...],
  "status": "degraded",
  "trace": {
    "fallbacks": ["rerank"],
    "errors": [{"stage": "rerank", "cls": "HTTPStatusError", "code": 429}]
  }
}
```

Возвращается retrieval order, `nDCG@50` падает на ~0.06 (см. [11_experiments.md](11_experiments.md)).
Но никаких 500, сервис продолжает отвечать.

### 12.10.6 Локальный ≠ hidden (§12.6)

Гипотетический пример: hidden-тест содержит вопрос «обсуждали ли мигратор БД?»,
а в хакатонских чатах такого не было.

Локально: система учится отвечать на technical product questions
хорошо (score 0.9694).

На hidden: если распределение сильно другое (больше HR-вопросов, product-
обсуждений, no-answer), наш local boost по `signal_terms` (цифры, технические
символы) не работает — эти сигналы редки в non-technical чате.

Митигация: recall-safe architecture (широкий prefetch, soft boost, graceful
fallback) — даже если все эвристики впустую, retrieval-only даёт базовый
Recall@50 на уровне hybrid BM25+dense.
