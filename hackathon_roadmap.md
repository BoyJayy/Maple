# Полная roadmap от текущего состояния до сдачи хакатона

Ниже уже нормальная, полная версия с учетом всего, что у нас теперь есть:

- ТЗ
- текущее состояние baseline
- Swagger по `index` сервису
- Swagger по `search` сервису
- то, что сейчас baseline уже поднимается, `GET /health` живой
- и то, что текущий `POST /index` по сути отдает почти заглушечный результат, а поиск у тебя ранее возвращал пустоту

---

# 1. Где мы находимся сейчас по факту

## 1.1 Что уже есть

У нас уже есть два сервиса:

- `index-service`
- `search-service`

И по API сейчас видны такие контракты.

### Index service
- `GET /health`
- `POST /index`
- `POST /sparse_embedding`

### Search service
- `GET /health`
- `POST /search`

---

## 1.2 Что видно по текущему baseline

### `GET /health`
Работает. Это хорошо, значит контейнер и FastAPI в целом поднимаются.

### `POST /index`
По текущему execute / example видно, что baseline сейчас формирует очень примитивный результат вида:

```json
{
  "results": [
    {
      "page_content": "string\nstring",
      "dense_content": "string\nstring",
      "sparse_content": "string\nstring",
      "message_ids": ["string"]
    }
  ]
}
```

Это почти наверняка означает не «реально умный chunking», а очень базовую схему:
- берет сообщения
- как-то склеивает текст
- отдает один или несколько примитивных чанков без нормальной логики контекста

### `POST /sparse_embedding`
Контракт есть, но по swagger видно только форму запроса/ответа. Значит sparse-часть пока рассматриваем как технически рабочую, но не оптимизированную.

### `POST /search`
Контракт богатый, и это очень важно. По search API видно, что на вход приходят не только:
- `question.text`

но и дополнительные сигналы:
- `asker`
- `asked_on`
- `variants`
- `hyde`
- `keywords`
- `entities.people`
- `entities.emails`
- `entities.documents`
- `entities.names`
- `entities.links`
- `date_mentions`
- `date_range`
- `search_text`

Это значит, что baseline, который ищет только по `text`, почти гарантированно слабый. Основной буст качества здесь будет не просто от «подключить dense+sparse», а от правильного использования всех этих полей.

---

## 1.3 Главный вывод по текущей точке

Сейчас состояние такое:

- инфраструктура есть
- API контракты есть
- сервисы поднимаются
- baseline индексатор слишком примитивный
- baseline поиск почти точно недоиспользует входной enriched question
- значит главная задача — не «написать сервисы с нуля», а заменить слабую retrieval-логику на сильную, не ломая контракт

# 3. Главная стратегия решения

Если совсем по сути, то задача распадается на две части.

## 3.1 `/index` отвечает за recall
Именно здесь ты решаешь:
- как разбить сообщения на чанки
- какой контекст попадет в один chunk
- какие тексты уйдут в dense
- какие тексты уйдут в sparse
- какие `message_ids` будет покрывать один результат

Если `/index` слабый, то даже сильный `/search` уже не спасет.

## 3.2 `/search` отвечает за ranking и добор recall
Здесь ты решаешь:
- как построить query из enriched question
- как использовать dense retrieval
- как использовать sparse retrieval
- как объединять кандидатов
- как применять reranker
- как учитывать даты, автора вопроса, entities и прочие сигналы

Если `/search` слабый, правильные сообщения могут находиться, но будут стоять слишком низко.

---

# 4. Целевое состояние решения

К концу разработки у тебя должно быть не «baseline чуть-чуть лучше», а вот такое решение.

## 4.1 Index service
Должен:
- стабильно принимать батчи сообщений
- корректно собирать чанки с учетом thread и времени
- формировать осмысленные `page_content`, `dense_content`, `sparse_content`
- возвращать хорошие `message_ids` покрытия

## 4.2 Search service
Должен:
- использовать все полезные поля `question`
- искать hybrid-способом: dense + sparse
- делать merge кандидатов
- rerank-ить top-N
- выдавать итоговый top-50 по релевантности

## 4.3 Система в целом
Должна:
- укладываться в лимиты по времени
- работать без интернета
- не ломать API контракты
- проходить полную проверку

# Этап 1. Нормальный Index pipeline вместо примитивного baseline

## Цель
Сделать так, чтобы индексатор создавал чанки, которые реально можно находить.

## 1.1 Сбор входных сообщений
Вход index-сервиса идет двумя кусками:
- `overlap_messages`
- `new_messages`

Нужно сразу принять правило:
- `overlap_messages` — это не мусор, а контекст
- `new_messages` — это новые сообщения, которые нужно проиндексировать
- chunking должен учитывать оба массива вместе

Базовая сборка:

```python
messages = overlap_messages + new_messages
```

Но этого мало.

## 1.2 Нормализация сообщений
Перед chunking каждое сообщение нужно нормализовать в внутреннюю структуру. Для каждого сообщения выделить:
- `id`
- `thread_sn`
- `time`
- `sender_id`
- очищенный `text`
- `file_snippets`
- упоминания
- признаки:
  - `is_system`
  - `is_hidden`
  - `is_forward`
  - `is_quote`

Также желательно вытащить дополнительный текст из:
- `parts`
- `file_snippets`
- `mentions`

## 1.3 Фильтрация мусора
Нужно аккуратно отфильтровать то, что почти наверняка вредит качеству.

Жестко убирать:
- `is_hidden == true`
- пустой текст и пустой файл-фрагмент
- чисто системные сообщения без поисковой ценности

Осторожно убирать:
- слишком короткие “ок”, “да”, “спасибо”
- forward/quote только если там реально нет нового текста
- member_event, если оно не несет полезного сигнала

Важно: нельзя бездумно выкинуть все короткие сообщения, потому что иногда ответ бывает типа:
- ссылка
- имя
- версия
- дата
- номер релиза

Поэтому фильтрация должна быть не тупая по длине, а по полезности.

## 1.4 Сортировка
После фильтрации все сообщения должны идти в стабильном порядке:

```python
messages.sort(key=lambda m: (m.time, m.id))
```

Это критично для воспроизводимого chunking.

## 1.5 Новый chunking: не по количеству сообщений, а по смыслу
Вот тут baseline обычно и сыпется.

### Плохой вариант
- просто по 5 сообщений подряд

### Нормальный вариант
Чанк строится на основе трех осей:

#### а) thread-aware
Если есть `thread_sn`, то сообщения одного треда должны иметь высокий шанс попасть в один chunk.

#### б) time-aware
Если между двумя сообщениями слишком большой разрыв по времени, лучше разрезать chunk.

#### в) size-aware
Нельзя делать слишком маленькие и слишком большие чанки.

## 1.6 Практическая стратегия chunking
Нормальная стартовая стратегия такая.

### Для thread-сообщений
- сначала группировать по `thread_sn`
- внутри треда сортировать по времени
- собирать chunk из последовательных сообщений треда, пока:
  - не превышен лимит по токенам
  - не произошел слишком большой time gap

### Для нетредовых сообщений
- строить временные окна по последовательным сообщениям
- склеивать сообщения, близкие по времени и теме

### Добавлять controlled overlap
Чтобы не терять контекст на стыке, можно делать overlap на:
- 1–2 сообщения
- или 15–20% чанка

Но overlap должен быть ограниченным, иначе:
- раздуется индекс
- появится куча дубликатов
- rerank станет слабее

## 1.7 Целевые размеры чанков
Стартовый ориентир:
- нижняя граница: примерно 80–120 токенов
- рабочий диапазон: 150–350 токенов
- верхняя граница: примерно 450–550 токенов

Для коротких, но ценных сообщений допускаются исключения.

## 1.8 Что должно попасть в `page_content`
`page_content` — это текст, который потом важен для rerank и общего контекста. Он должен быть читаемым и полным.

Хороший шаблон:

```text
ЧАТ: {chat.name}
ТИП ЧАТА: {chat.type}
CHAT_ID: {chat.id}
THREAD_SN: {thread_sn or "no_thread"}

[2025-10-20 14:31:00 | sender_1]
текст сообщения 1

[2025-10-20 14:32:10 | sender_2]
текст сообщения 2

Вложения / file snippets:
...
```

Что это дает:
- модель видит не только сырой текст
- модель видит структуру диалога
- упрощается rerank

## 1.9 Что должно попасть в `dense_content`
`dense_content` — это не обязано быть копией `page_content`.

Здесь нужен текст, оптимизированный под semantic retrieval.

Нужно:
- убрать лишний шум
- оставить смысл
- убрать лишнюю служебность
- оставить имена, продукты, версии, документы, ссылки, важные даты

Хороший состав:
- краткий контекст чата
- основной текст сообщений
- `file_snippets`
- упоминания имен/сущностей
- без лишних служебных заголовков, если они засоряют эмбеддинг

## 1.10 Что должно попасть в `sparse_content`
`sparse_content` должен быть ориентирован на keyword retrieval.

Сюда полезно включать:
- исходный текст сообщений
- `file_snippets`
- имена людей
- email’ы
- названия документов
- внутренние имена сервисов и продуктов
- ссылки
- date expressions
- нормализованные формы слов, если делаешь такую обработку

Идея простая:
если пользователь спрашивает буквально, sparse должен зацепиться за exact/surface form.

## 1.11 Как формировать `message_ids`
Это очень важно. В `message_ids` надо класть ровно те сообщения, которые покрывает chunk.

Нельзя:
- класть только первый id
- класть случайный id
- терять часть покрываемых сообщений

Потому что итоговая метрика оценивает попадание по message id.

## 1.12 Локальная проверка index
После переписывания `/index` проверить:
- сколько средне выходит чанков на чат/батч
- нет ли слишком много дублей
- не слишком ли длинные тексты
- несут ли `dense_content` и `sparse_content` разный полезный сигнал
- совпадают ли `message_ids` с содержимым чанка

## Критерий завершения этапа
Индексатор:
- не заглушечный
- стабильно строит осмысленные чанки
- хорошо покрывает диалоги и треды
- не теряет message ids

# Этап 2. Сильный sparse embedding

## Цель
Сделать sparse retrieval реально полезным, а не формальным.

## 2.1 Что должно делать `/sparse_embedding`
Эндпоинт принимает список текстов:

```json
{"texts": ["...", "..."]}
```

И возвращает sparse-векторы совместимого формата.

По ТЗ это будет использоваться:
- при индексации
- при поиске

Значит реализация должна быть:
- стабильной
- быстрой
- воспроизводимой
- локальной, без интернета

## 2.2 Практический вариант реализации
Стартовый вариант:
- BM25 / fastembed / аналогичная библиотека, которую можно упаковать внутрь контейнера

Важно проверить:
- модель и веса реально лежат внутри образа
- контейнер не требует интернета
- latency приемлема и на индексации, и на поиске

## 2.3 Что может бустить sparse качество
Если библиотека и схема позволяют, для sparse полезны:
- термы из `file_snippets`
- термы из ссылок
- имена людей
- названия документов
- названия продуктов / систем
- упоминания версии, даты, релиза, номера тикета

## Критерий завершения этапа
Sparse реально дает полезных кандидатов, а не просто существует ради контракта.

# Этап 3. Полная переработка Search pipeline под enriched question

## Цель
Использовать `POST /search` по-настоящему, а не только `question.text`.

## 3.1 Что сейчас является главным преимуществом search API
В search API уже есть готовая «обогащенная» форма вопроса. Это огромный подарок. Не нужно самим генерировать из сырого текста все признаки — часть уже пришла.

Вопрос содержит:
- основной текст
- варианты перефразировки
- HyDE-подсказки
- keywords
- entities
- date mentions
- date range
- search_text
- asker
- asked_on

Если baseline это не использует, он просто сливает потенциальное качество.

## 3.2 Построение query representation
Нельзя делать один-единственный query string и на этом успокаиваться.

Нужно строить несколько представлений запроса.

### Основной semantic query
Лучший кандидат:
- `search_text`, если он непустой
- иначе `text`

### Дополнительные semantic expansions
- `variants`
- `hyde`

### Keyword query
- `keywords`
- surface forms из `entities`
- возможно дата-термы

### Метаданные запроса
- `asker`
- `asked_on`
- `date_range`

## 3.3 Рекомендуемая структура query builder
Собираем несколько видов запросов.

### Dense query
Основной dense запрос:
- `search_text + text + лучшие variants`

HyDE можно:
- добавлять не в один общий суп из всего подряд
- а либо отдельными dense retrieval запросами
- либо ограниченно смешивать с основным текстом

Иначе есть риск размазать embedding.

### Sparse query
Основной sparse запрос строить из:
- `search_text`
- `keywords`
- entity names
- document names
- emails
- links
- явных date mentions

### Filters / boosts
Отдельно выделить:
- `date_range`
- сигналы по `asker`
- сигналы по людям / email / документам

# Этап 4. Dense retrieval

## Цель
Получать семантически близкие кандидаты даже при слабом lexical overlap.

## 4.1 Вызов dense embedding API
Dense модель дается внешним API. Нужно:
- стабильно вызывать ее
- учитывать таймауты
- не делать лишних вызовов
- батчить где возможно

## 4.2 Какой текст отправлять в dense на search
Не всегда лучший dense query — это просто `question.text`.

Приоритет примерно такой:
1. `search_text`
2. `text`
3. 1–3 лучших `variants`
4. аккуратно — `hyde`

Нельзя бездумно склеить всё подряд, иначе семантический вектор станет грязным.

## 4.3 Сколько dense кандидатов брать
Стартово:
- prefetch dense: 50–150
- рабочая точка для тюнинга: 80–120

Если кандидатов слишком мало — теряешь recall.
Если слишком много — растет шум и время rerank.

# Этап 5. Sparse retrieval

## Цель
Ловить точные совпадения по словам, именам, документам, версиям, ссылкам и датам.

## 5.1 Какой текст отправлять в sparse на search
Строить sparse query из:
- `search_text`
- `keywords`
- `entities.people`
- `entities.emails`
- `entities.documents`
- `entities.names`
- `entities.links`
- `date_mentions`

Это особенно важно для вопросов типа:
- “кто писал про релиз Go”
- “где кидали ссылку на документ”
- “что писали про приказ”
- “кто присылал email”
- “что было 13 октября”

## 5.2 Сколько sparse кандидатов брать
Стартово:
- 50–150

Дальше тюнить совместно с dense.

# Этап 6. Merge dense и sparse кандидатов

## Цель
Собрать один сильный candidate set перед rerank.

## 6.1 Почему нельзя просто concat
Если тупо склеить dense и sparse:
- будут дубли
- score разных систем несопоставимы
- слабые кандидаты засорят rerank

## 6.2 Что делать правильно
Нужно:
- дедуплицировать по идентификатору point/chunk
- нормализовать dense и sparse score
- объединять через weighted score

Стартовая формула:

```text
final_prefetch_score = 0.55 * dense_norm + 0.45 * sparse_norm
```

Потом тюнить:
- `0.6 / 0.4`
- `0.5 / 0.5`
- `0.65 / 0.35`

## 6.3 Дополнительные boosts на merge этапе
Еще до rerank можно слегка бустить chunk, если:
- есть совпадение по entity
- есть совпадение по document title
- совпадает email
- есть попадание в date range
- sender совпадает с `asker` или тесно связан с ним
- chunk из релевантного thread

Но boosts должны быть аккуратными, иначе сломаешь ranking.

# Этап 10. Rerank pipeline

## Цель

## 10.1 Что отдается в reranker
В reranker должен идти не просто сырой текст, а хороший pair:
- query
- document

### Хороший query для rerank
Не обязательно тот же, что для dense.
Чаще всего лучший rerank query:
- `search_text`, если есть
- иначе `text`

Лучше всего `page_content`, потому что там полный контекст.

## 10.2 Сколько кандидатов отправлять в rerank
Стартовая зона:
- 30–100 кандидатов

- prefetch 80–150
- rerank top 40–80

## 10.3 Что может дополнительно улучшать rerank
После rerank можно сделать небольшой post-processing:
- collapse near-duplicates
- ограничить повторяющиеся почти одинаковые чанки из одного и того же места
- следить, чтобы в top не было 10 одинаковых overlap-результатов

# 10. Что обязательно добавить в архитектуру, чтобы roadmap была полной

Ниже то, что обязательно стоит зафиксировать отдельно, чтобы команда не уперлась в скрытые дыры уже во время реализации.

## 10.1 Полный pipeline индексации до Qdrant

Важно помнить: `POST /index` сам по себе не завершает индексацию.

Полный pipeline должен выглядеть так:

1. Вызвать `POST /index`
2. Получить список chunk'ов:
   - `page_content`
   - `dense_content`
   - `sparse_content`
   - `message_ids`
3. Посчитать dense embeddings для `dense_content`
4. Посчитать sparse embeddings через `POST /sparse_embedding` для `sparse_content`
5. Сформировать payload и metadata для каждой точки
6. Сделать upsert point'ов в Qdrant

То есть индексатор в реальной системе - это не только chunking, но и полная сборка point'ов для коллекции.

Если этот шаг не прописан явно, можно получить хороший `/index`, но при этом так и не построить рабочий retrieval.

## 10.2 Схема point'а в Qdrant

Нужно заранее зафиксировать, что именно хранится в каждой точке коллекции.

Минимально разумная схема:

```json
{
  "id": "stable_chunk_id",
  "vector": {
    "dense": "...",
    "sparse": "..."
  },
  "payload": {
    "page_content": "...",
    "dense_content": "...",
    "sparse_content": "...",
    "metadata": {
      "chat_name": "...",
      "chat_type": "...",
      "chat_id": "...",
      "chat_sn": "...",
      "thread_sn": "...",
      "message_ids": ["..."],
      "start": "...",
      "end": "...",
      "participants": ["..."],
      "mentions": ["..."],
      "contains_forward": false,
      "contains_quote": false
    }
  }
}
```

### Что особенно важно

- `message_ids` обязательно должны лежать в metadata
- `start` и `end` желательно хранить для time-aware filtering/boosting
- `participants` и `mentions` пригодятся для entity-aware retrieval
- `thread_sn` пригодится для thread-aware boosting и анализа дублей

## 10.3 Как генерировать stable id для chunk'а

Point id в Qdrant не должен быть случайным.

Хороший вариант:
- хэш от:
  - `chat_id`
  - `thread_sn`
  - первого `message_id`
  - последнего `message_id`
  - текста chunk'а или его нормализованной формы

Это даст:
- стабильный upsert
- воспроизводимость
- меньше проблем с дублями

## 10.4 Кто именно отвечает за upsert

Нужно зафиксировать это архитектурно, иначе на командной работе начнется путаница.

Надо явно определить:
- какой модуль вызывает `POST /index`
- какой модуль вызывает dense embeddings API
- какой модуль вызывает `POST /sparse_embedding`
- какой модуль собирает point'ы
- какой модуль делает upsert в Qdrant

Лучше мыслить так:
- `index-service` отвечает за chunk semantics
- отдельный indexing orchestrator отвечает за превращение chunk'ов в Qdrant points

# 11. Что обязательно добавить в search roadmap

## 11.1 Безопасный стартовый baseline для merge

Не стоит сразу переусложнять merge dense и sparse score.

Безопасный стартовый порядок такой:

1. dense retrieval
2. sparse retrieval
3. объединение кандидатов через rank-based fusion
4. rerank top-N
5. post-processing

### Почему это важно

Score разных retrieval-веток часто плохо сопоставимы.

Поэтому:
- сначала лучше использовать rank-based fusion / RRF
- а уже потом, если локальная evaluation покажет пользу, пробовать нормализацию score и weighted blending

Иначе можно потратить много времени на красивую, но нестабильную математику merge.

## 11.2 Отдельная сборка финального ответа

После retrieval и rerank работа еще не закончена.

Нужно явно добавить этап final assembly:

1. взять reranked chunk results
2. убрать почти одинаковые overlap-chunks
3. убрать сильные near-duplicates из одного и того же места
4. аккуратно собрать итоговый список `message_ids`
5. ограничить выдачу top-50

## 11.3 Дедуп не только chunk'ов, но и `message_ids`

Это отдельный важный пункт.

Если overlap сделан хорошо, почти наверняка появятся:
- соседние чанки с похожим текстом
- повторяющиеся `message_ids`
- одинаковые ответы, пришедшие из dense и sparse веток

Значит надо отдельно решить:
- дедуп по point id
- дедуп по набору `message_ids`
- дедуп по пересечению покрытия сообщений

Иначе top-50 может визуально быть длинным, а фактически содержать мало уникальной полезной информации.

## 11.4 Что считать сильным post-processing

Хороший post-processing:
- не ломает ranking
- не выкидывает редкие, но ценные результаты
- режет только явные дубли
- не дает одному треду занять весь top

Практически это значит:
- ограничить число почти одинаковых chunk'ов из одного thread/window
- оставлять лучший из почти идентичных overlap-кандидатов
- следить за разнообразием top-results

- more advanced merge

Так ты не потеряешь контроль над качеством и быстрее поймешь, что реально помогает метрикам.

## Шаг 5. Вести эксперименты как таблицу гипотез

Для каждой версии фиксировать:
- что поменяли
- зачем поменяли
- recall@50
- ndcg@50
- latency
- побочные эффекты

Это особенно важно в финальные дни, когда очень легко начать тюнить хаотично.

# 13. Самые частые скрытые риски

Чтобы команда не наступила на типовые грабли, полезно прямо зафиксировать вот эти риски.

## 13.1 Сильный `/index`, но слабый ingestion

Можно хорошо строить чанки, но:
- не делать upsert
- делать плохой payload
- терять metadata
- генерировать нестабильные point id

И тогда search будет казаться слабым, хотя проблема на самом деле раньше.

## 13.2 Сильный retrieval, но плохой final assembly

Можно хорошо находить кандидатов, но испортить итог:
- дублями
- повторяющимися `message_ids`
- слишком длинным overlap-tail
- засорением top-50 одним и тем же куском диалога

## 13.3 Переусложнение слишком рано

Самая частая ошибка:
- сразу делать сложные boosts
- сразу делать хитрую формулу merge
- сразу смешивать все поля `question` без контроля

Чаще выигрывает более простой, но устойчивый pipeline, который затем тюнится по метрикам.

## 13.4 Недооценка enriched question

Если проигнорировать:
- `search_text`
- `keywords`
- `entities`
- `date_range`

то будет потеряно большое количество уже готового сигнала, который в API фактически выдали бесплатно.
